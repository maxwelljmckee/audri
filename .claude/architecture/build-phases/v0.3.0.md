# Build plan — v0.3.0 (private-release polish)

**Status:** in planning, opened 2026-05-12. Phase reframe from the original "installable plugins + connectors + dreaming" theme — see "Reframe" section below for the rationale and the items that moved back to backlog.

The phase that takes Audri from "MVP + autonomic loop + usage tracking" to **ready-for-private-release**: a select few trusted testers can be onboarded. Public-release infrastructure (waitlist, admin, transactional email) explicitly stays deferred.

Instead, deeply refine the services we already have (Notes, Todos, Research) + introduce the two new tiles needed for the private-release surface (**Automations** + **Storage / document uploads**), with the Agents tile getting its own full pass. Then make them feel cohesive via cross-linking.

## Phase scope

The phase organizes around **three sequenced steps** plus two locked substrate items:

1. **Backend service quality** — Live Agent + Ingestion + Research prompt + behavior refinement.
2. **Per-tile UX polish (sequential)** — Notes → Todos → Automations → Storage → Research → Agents. Each tile gets a focused pass.
3. **Interconnection** — cross-linking + reingestion + source attachment, tying the polished tiles into a coherent surface.

Plus, threaded through:
- **Archive substrate** (`archived_at` on `wiki_pages`) — locked, prereq for Dreaming automation.
- **Hard spending-cap enforcement** — locked, backend-focused, composes with v0.2.1's Usage substrate.

## In scope

### Track A: Backend service quality

#### A1. Live Agent

The Live Agent stays a **read-only consumer** of the wiki + artifact graph. **Ingestion is the single source of truth for every commit.** No `create_todo` / `create_note` mid-call tools; the trust gap ("Audri said she'd write that down but didn't") closes via better intent recognition in ingestion + a pending-capture UI affordance on the call screen.

1. ⏺️ **Read tools — `search_research`, `fetch_research`, `search_storage`, `fetch_storage`.** Join the existing `search_wiki` / `fetch_page`. Live agent can pull recent artifacts on demand: "what did we research about Italian restaurants?" → `search_research` → `fetch_research(id)` → spoken summary. Each tool: server endpoint runs under user's JWT (RLS scopes results); usage_events row written per call; tool registration in the live-prompt's tool palette.

2. ⏺️ **Pending-capture UI indicator on the call screen.** When the user expresses an intent ("write this down", "remind me to..."), a chip on the call screen acknowledges "Audri noted: 1 item will land after this call." Closes the trust gap without enabling mid-call mutations. Source signal: the live agent emits a `note_pending(summary)` tool call that purely renders UI — no DB writes. Audited via the existing transcript record.

3. ⏺️ **Tour-guide capability — UX awareness + coaching.** Reframed from the original "Live Agent ↔ UI capability parity" backlog entry. The live agent gains knowledge of how the app is structured (tiles, navigation, feature locations), can coach the user through using features ("the dream we just generated lives under your Agents tile, want me to walk you through it?"), and lays the substrate for future programmatic UI control (push-screen tools). v0.3.0 scope: prompt-side awareness + coaching cues, NO programmatic-UI tools yet — that's V1+. **App-map generation (hybrid):** registry contributes structured pieces (each `pluginRegistry` entry's name, capability description, icon, navigation slug); narrative wraps them with "how to navigate" prose that doesn't fit registry metadata. Prevents drift when plugins evolve while keeping flexibility for human-authored guidance. **Caching pairing:** the app map is exactly the stable prefix Gemini's explicit caching rewards — pairs with the "Gemini explicit caching for ingestion scaffolding" backlog item, ideally landed in a single caching pass.

4. ⏺️ **Live abort tool — mid-action cancellation.** Tool the agent invokes when the user pivots mid-call ("never mind, let's talk about X instead") to cleanly cancel an in-progress operation (research task, open UI surface, draft writes). Pairs with future Interactive Call Surface but lands here for the agent_task-cancellation case at minimum.

5. ⏺️ **Agent-driven call-end tool — `end_call(reason)`.** Tool registration so the agent can hang up after closing pleasantries. Closes the awkward dead-air gap after "alright, talk soon!" Mirrors POST `/calls/:id/end` with `end_reason='agent_ended'`.

6. ⏺️ **Proactive call-end (adaptive guardrail).** Most calls under the terse-command reframe will be 30-second imperative loops, not 20-min brainstorms. Timer adapts: short cap (~2 min) by default; extends to longer cap when conversation patterns signal genuine brainstorm/long-form work. Hard cap triggers definitive close-out turn. Per-user override deferred to V1+. Depends on #5 (the mechanism to hang up). *(Was #7 in the kickoff queue — renumbered post-Q11 removal.)*

7. ⏺️ **Ongoing prompt iteration.** Continued refinement of system-prompt's "Reading the moment", "Bring something to the conversation", "Terse after delegation" sections based on field-test feedback. Threading the terse-command reframe through here: most calls expected to be short imperative loops, but don't overcorrect — preserve graceful handling of the rare long-form brainstorm. No hard milestone; iterate until private-release confidence is reached.

70. ⏺️ **Text-chat mode.** Parallel I/O channel to the live call — same agent foundation, different I/O plumbing. UI: chat tab/sheet alongside the call FAB. Composer reads a `mode='text'` flag at call start and prunes voice-specific scaffolding (brevity-for-spoken-audio, audio-modality tool-selection clauses, etc.). Transcripts share shape with voice transcripts so post-call ingestion pipeline is unchanged. Voice-first users can't always talk (meetings, noisy environments); text mode covers that. *(Promoted from backlog P0 entry on 2026-05-13.)*

71. ⏺️ **Incognito calls — no ingestion.** New `call_type='incognito'` flag opts the call out of post-call ingestion entirely — neither user-scope fan-out nor agent-scope ingestion runs. Transcript persists for the user's own retrieval. UX: toggle on the call screen with clear visual treatment (dimmed orb, indicator chip) so the user always knows they're in this mode. Distinct preload posture: skip recent activity + wiki preload; distinct prompt clause ("ephemeral mode; don't reference their wiki, don't extract claims, treat this as a closed exchange"). Use cases: venting, sensitive emotional content, exploratory thinking the user doesn't want pinned. *(Promoted from backlog P1 entry on 2026-05-13.)*

79. ⏺️ **Chunked ingestion for large documents.** When a doc's extracted text exceeds a threshold (~50k tokens / ~150 pages worth), split into chapter-sized chunks (~25-30k tokens each) and run the Flash + Pro fan-out per chunk, then synthesize the chunk outputs into a single canonical source page (one section per chunk, or merge-where-coherent). **Why:** uncapped doc ingestion can hang indefinitely — Plato's Republic (297k input tokens, ~400 pages) ran past 17 min on the first attempt 2026-05-15 before being manually aborted, with no native timeout because undici's `bodyTimeout` is per-chunk (a slow Gemini token stream never trips it). Beyond the hang risk, the cost-per-doc is dominated by input tokens (~$0.60 for Plato vs. ~$0.05 for an article); chunking unlocks (a) parallel chunk processing for faster wall-clock, (b) input prompt caching across chunks for cost amortization, (c) graceful partial-failure (if chunk 3/8 fails, retry just that chunk; the other 7 already landed). **Scope at execution:** (i) chunking heuristic — split on chapter/section boundaries when extractable, else sliding-window with overlap; (ii) per-chunk Pro fan-out producing an intermediate per-chunk result; (iii) synthesis step that merges into the final source page (could be a final Pro call OR deterministic concatenation depending on quality tradeoff); (iv) commit step writes ONE source page (still per the "one canonical page" rule) with sections derived from chunk outputs; (v) per-chunk usage_events for cost attribution. **Pairs with:** prompt caching (backlog) — chunking amplifies caching benefit since the static system prompt is re-used across all chunks; the wall-clock timeout guard (also new — see backlog) which would catch any single-chunk runaway. Source: 2026-05-15 dogfood incident.

### Track E: Dreams (3-tier synthesis substrate)

Added 2026-05-15. Replaces the existing `agent_open_items` substrate with a 3-tier sleep-inspired model (Light / REM / Deep, Deep deferred to v0.4). The dominant design insight: dreams are a **proposal substrate, never a write surface** — integration into the wiki happens only through live-call discussion + the existing transcript ingestion pipeline. AI synthesis lands in `dreams`; the user discusses, ingestion writes. No silent edits, no separate apply path, hardened-codepath reuse.

Closely coupled to the v0.3 doc/url ingestion tracks (B2 + the url-source pipeline): docs/URLs now produce a CONTAINED output (source page + minimal cross-link stubs only). Cross-page enrichment from a new doc is REM's job — surfaced to the user via Dreams + the next call. The "automatic feeling" of doc ingestion shifts from "silent multi-page rewrite at ingest time" to "agent connects the dots during your next call." See `project_dreams_architecture` memory + `project_ingestion_philosophy` for the philosophy.

Sequencing within the track is roughly top-down — D1 (substrate) gates everything else; D5 (in-call surfacing) gates the engagement flow; D6 (mobile UX) can land in parallel with D7. 

90. **D1: Migrate `agent_open_items` → `dreams`.** Rename table; rename enums (`agent_open_item_kind` → `dream_kind` with new `insight` value, `agent_open_item_status` → `dream_status` — existing status values stay). Add `tier dream_tier NOT NULL` column with values `light` / `rem` / `deep`. RLS policies, realtime publication, mobile RxDB schema all follow the rename through. Migration uses `ALTER TYPE` + `RENAME TABLE`; backfill `tier='light'` for existing rows. Touches: `packages/shared/src/db/schema/`, `apps/server/src/`, `apps/worker/src/`, `apps/mobile/lib/rxdb/`. Source: 2026-05-15 design discussion.

91. **D2: Light rebrand — current post-call agent-scope pass becomes Light dreaming.** Today's `apps/worker/src/ingestion/agent-scope.ts` produces both agent-scope wiki writes AND open-items rows. Under the new model, the open-items rows become Light-tier dreams. Code change is minimal — same pipeline, set `tier='light'` on the new dreams inserts. Naming + comments updated to reflect the Light-tier framing. Doesn't change runtime behavior; preserves existing semantics under the new conceptual umbrella.

92. **D3: REM handler + nightly schedule.** New `dreaming-rem` graphile task. Recurring trigger via `dispatch-recurring.ts` (existing infra for automations). Handler: load N days of recent context (transcripts + new wiki pages + new docs + new artifacts), Pro-tier synthesis, output `insight`-kind dreams capturing cross-source connections. Respects spend cap (silent skip when over, same pattern as ingestion `skipped_over_cap`). No on-demand trigger for MVP — user can ask the live agent to "process this now" if they want immediate handling. Source: `project_dreams_architecture` 3-tier model.

93. **D4: Doc + URL ingestion Path B lock-in.** Already partly landed via the 2026-05-15 prompt rewrites — `apps/worker/src/uploads/fan-out.ts` and `apps/worker/src/url-sources/fan-out.ts` now enforce contained output (source page + minimal cross-link stubs, no enrichment to existing pages). This item tracks dogfooding the new behavior on real docs/URLs and confirming no cross-page leakage. Once D3 lands, manually trigger one doc through the pipeline → verify nothing wrote to existing pages → confirm next-night REM produces sensible proposals.

94. **D5: Live-call preload extension + system-prompt addition.** Extend `apps/server/src/calls/preload.ts` with a slim "recent unsurfaced dreams" block (3-5 dreams, ranked by recency + topic affinity to other recent activity). Add a section to `apps/server/src/calls/system-prompt.ts` instructing the live agent on WHEN + HOW to surface dreams: proactively at natural conversation openings (especially morning calls / first-of-day), gated by relevance to the unfolding conversation, framed as offers rather than impositions ("Want to talk about something that came up?"). Surfaced dreams flow into the conversation naturally; user's discussion becomes transcript content that standard ingestion processes.

95. **D6: Mobile Dreams feed UX.** Minimal first cut — a Plugin tile or surface on the home grid that renders pending dreams as a list. Per-row actions: open detail / accept / dismiss / "discuss next call" (just marks `surfaced`, leaves engagement to the live agent). Tier badge so the user can distinguish Light (per-call reflections) from REM (overnight synthesis). RxDB sync of the renamed table is already in place from D1. Low-chrome; iterate later if usage warrants more affordance.

96. **D7: Engagement mechanic.** Decide: tool-call `mark_dream_engaged(id)` invoked by the live agent during conversation (deterministic, requires the agent to remember to call it), OR post-hoc match in the worker (the ingestion pipeline, after writing transcript content, scans pending dreams for content overlap and marks engagement automatically). Trade-offs: tool-call is precise but adds agent cognitive load; post-hoc match is lower-friction but probabilistic. Lean post-hoc match for MVP; revisit if matching accuracy is poor.

**Deferred to v0.4:**

## Mid-phase decisions log (2026-05-14)

While building C4 P1 substrate:

- **"Documents" → "Storage / uploads" rename.** Original framing was too narrow for the image/audio kinds that will follow. Surface-wide rename: tile = Storage, primary table = `uploads`, bucket = `audri_storage`. URL ingestion is a sibling concern, not an upload kind.
- **URL-as-source pipeline (#80) split out as parallel sibling to #27.** There's no Storage object server-side for a URL — separate table, separate ingestion pipeline. Same Path B junction shape so the mobile feed merges cleanly via a `family` discriminator.
- **Path B junction pattern for attachments.** Each `(upload | url_source, page)` attachment gets its own row with its own extraction + ingestion status lifecycle. Enables multi-attach (one upload → many pages) without conflating status, and decouples re-ingestion per attachment.
- **URL kinds shipped inline:** `web_article` (Readability + JSDOM), `pdf` (pdf-parse), `reddit_thread` (public `.json` API). YouTube / Twitter / podcast / generic-OG metadata kinds → backlog.
- **`webidl-conversions` pin via `pnpm.overrides`.** Mobile Hermes can't survive `webidl-conversions@8`'s unguarded `SharedArrayBuffer.prototype` access at module-init. Root `package.json` now pins `webidl-conversions: ^7.0.0` (which kept its `typeof` guard). The mobile `metro.config.js` worker-only blocklist (`jsdom`, `@mozilla/readability`, `pdf-parse`, `mammoth`, `undici`) stays as hygiene but is no longer load-bearing. **Backlog:** the `nodeLinker: hoisted` setup makes this class of failure recurring — any future worker dep with an unguarded Node-only global at module-top will repeat the symptom. Structural fix (move worker out of workspace, or switch to `nodeLinker: isolated` + Metro symlink support) noted in backlog.
