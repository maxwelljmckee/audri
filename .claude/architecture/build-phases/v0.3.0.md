# Build plan — v0.3.0 (installable plugins)

The phase that introduces **installable plugins** as a generalizable pattern, ships the first three (Connectors, Dreaming, plus the Archive feature Dreaming depends on), and restores the dreaming line to the onboarding script. Where v0.2.0 turned Audri into a system that *grows its own knowledge*, v0.3.0 turns Audri's plugin registry into a *user-extensible surface* — the substrate that future specialist agents, V1+ plugins, and the eventual plugin marketplace will all build on.

## Phase scope

Three coherent tracks share one new architectural primitive:

1. **Installable-plugin substrate** — the primitive itself. Today every plugin in the registry is implicitly always-on. After this phase, plugins can be opt-in with a per-user install flow, an install-time hook that mutates user state (e.g. seeding pages), per-user cadence/configuration, and a dispatcher that consults installed-state before scheduling work.
2. **Connectors** — the OpenClaw Path 1 recommendation: wrap the standalone MIT-licensed CLIs (`gogcli` / `notcrawl` / `remindctl` / etc.) as plugin handlers, with per-user OAuth flows. First two installable connectors honor the v0.2 onboarding-script promise *"plug me into your email, schedule, and meeting apps."*
3. **Dreaming** — the weekly-synthesis pass we resolved in v0.2's DP-2. Bundled as the first installable plugin built on the substrate (track #1). Restores the dreaming line to onboarding; ships with the Archive feature it depends on.

Plus one parallel track that's a hard prereq for Dreaming and worth shipping standalone: **Archive feature** (`archived_at` timestamp, per `backlog.md`). Not a plugin — it's a wiki-substrate change applied universally.

## Why this is a coherent phase

The three tracks aren't independent — they share the install primitive, and Dreaming literally depends on Archive. The natural sequence:

1. Install substrate first (smallest, unblocks the rest).
2. Archive feature next (small, unblocks Dreaming's review-or-archive UX).
3. First connector end-to-end (validates the install pattern with real OAuth + artifact tables).
4. Dreaming (validates the install pattern with a scheduled, internally-driven plugin).
5. Second + third connector (parallelize once the pattern is proven).
6. Onboarding script revision restores the dreaming line and adjusts the connectors line as appropriate.

## In scope

### Installable-plugin substrate

1. ⏺️ **`user_plugins` table + plugin lifecycle** — schema for tracking which plugins each user has installed, with optional per-install configuration (cadence, etc.). Likely shape: `id`, `user_id`, `plugin_kind`, `installed_at`, `uninstalled_at`, `config jsonb`. Plugin registry gains an `installable: bool` flag and an `onInstall` / `onUninstall` hook contract.
2. ⏺️ **Dispatcher consults install state** — `agent_task_dispatch` (and any cron-driven scheduler) skips work for users without the relevant plugin installed.
3. ⏺️ **Mobile install/uninstall affordance** — UI surface to enable/disable plugins. Initially manifest-driven (built-in list); evolves toward marketplace-style discovery in V1+. Decision-point territory: where this surface lives in the plugin grid (a "+" tile? settings menu? both?). Defer detailed UX to phase planning.
4. ⏺️ **Capability-description gating** — the call-side prompt only describes installed plugins to the live agent. Otherwise Audri promises capabilities the user hasn't enabled.

### Archive feature (prereq for Dreaming)

5. ⏺️ **`archived_at` timestamp on `wiki_pages`** — per `backlog.md` Archive entry. Nullable column; null = active, timestamp = archived. Includes archive-aware queries: preload, `search_wiki`, fan-out targeting all exclude archived pages and their descendants (single recursive CTE for ancestry walk-up). Unarchive = `archived_at = NULL`. UI surface in Wiki plugin: archive/unarchive affordance per page, archived-pages browse view (excluded by default).

### Connectors track

6. ⏺️ **Connectors plugin family + first connector end-to-end** — OpenClaw Path 1 recommendation. Wraps standalone MIT-licensed CLIs (`gogcli` first — Gmail / Calendar) as plugin handlers in the worker. Per-user OAuth state stored in Postgres, projected into per-job env vars or temp keyring files when invoking the CLI. Per-connector artifact tables (`gmail_threads`, `calendar_events`) following the `research_outputs` pattern. Mobile UI: per-connector OAuth grant flow + artifact display via `useResearchOutputs`-style hooks. Studied (not lifted as a runtime dep): OpenClaw's plugin manifest format and capability ownership model. Per `openclaw-evaluation.md` recommendation: ~1–2 weeks per major connector once the pattern is in.
7. ⏺️ **Second + third connector** — `notcrawl` (Notion) and one of `remindctl` (Apple Reminders) / `imsg` (iMessage). Selection deferred to planning.

### Dreaming plugin

8. ⏺️ **Dreaming plugin (MVP design from DP-2 resolution, 2026-05-09)** — installable. On install: seeds top-level user-scoped `Dreams` page. Plugin handler runs the Light/REM/Deep phase split:
    - *Light:* scan recent activity (extracted_claims, agent-scope observations, recent wiki_sections, raw transcripts in a configurable window) for candidate items.
    - *REM:* score candidates using the six OpenClaw weights (frequency, relevance, query-diversity, recency, consolidation, conceptual-richness). Configurable weight tuning deferred to V1+; ship with OpenClaw defaults.
    - *Deep:* generate dream artifacts. Each dream/insight = its own `wiki_pages` row of `kind = 'dream'`, parent = `Dreams`. Promotion = parent_page_id mutation (no section rewrite). Archive = `archived_at` set (uses #5). Default unreviewed-dream auto-archive after 2 weeks (configurable).
    - *Cadence:* configurable per-user, default weekly. Lives in `user_plugins.config`.
    - *Promotion target picker:* MVP = Option A (agent suggests parent based on dream content + existing wiki, user confirms or overrides). V2+ = Option C (full agent-suggest + user-browse + override).
9. ⏺️ **`kind = 'dream'` enum addition** — small migration to `wiki_pages.kind`. Promotion mutates `kind` (`dream` → `concept` / `note` / `project` etc.) alongside `parent_page_id`.
10. ⏺️ **Dreams plugin UI surface** — installable plugin tile in mobile shell. Lists dreams under `Dreams`, exposes review-and-promote-or-archive interaction, surfaces dream content + the Light/REM/Deep provenance metadata (so the user can see *why* this dream surfaced).

### Onboarding script revision

11. ⏺️ **Dreaming line restored to onboarding script** — `apps/server/src/calls/system-prompt.ts → composeOnboardingScaffolding`. Restore the cut line: *"I can even Dream — that's where I mull over your notes in my sleep and come up with new synthesis and connections that I can surface to you."* Mirror to `specs/onboarding.md`. ## Out of scope / deferred (revisit at phase planning)

- **Plugin marketplace surface (XL effort)** — separate from the install pattern. The install pattern this phase builds is the *substrate* for a marketplace; the marketplace itself (third-party plugin discovery, user-submitted plugins, reputation/moderation) is V1+.
- **Configurable scoring weights for Dreaming** — ship with OpenClaw defaults in v0.3; tune via UI in V1+.
- **`DREAMS.md` Dream Diary as a separate artifact** — superseded by per-dream wiki pages (one-page-per-dream is a strict superset of the diary surface).
- **Cross-persona dreaming** (one persona reading another's agent-scope) — V1+, gated on whether multi-persona wiki sharing makes product sense at all.
- **Connectors beyond the first three** — `wacli`, `discrawl`, `Peekaboo`, `AXorcist`, etc. Backlog them; install-pattern proven by the first three opens the door.
- **Configurable Light/REM/Deep prompt customization per user** — power-user feature, V1+.
