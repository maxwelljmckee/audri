# Build plan — v0.1.1 (UX polish patch)

A patch-version cycle between MVP (0.1.0, shipped 2026-04-29) and the next feature wave. Focused entirely on mobile-side UX polish — the rough edges that became visible once the app was usable end-to-end on TestFlight. The cycle splits into two passes: a **visual-identity + core surfaces** pass (the priority items below — splash, auth, onboarding, home, call, plugin shell), and a **rough-edges** pass (the smaller backlog-derived items pulled forward into this patch).

Items below the priority block are copied from `backlog.md` so this doc is self-contained for the cycle. When an item ships, mark it ✅ here AND remove (or strikethrough + mark "shipped in 0.1.1") in the backlog. If we change our minds on scope mid-cycle, items can move back to `backlog.md` freely.

**Legend**

- ✅ — done
- ✴️ — partial / in progress
- ⏺️ — open / not started
- ⛔ — blocked

## In scope

### Priority — visual identity + core surfaces

The top-priority block. These items reshape the look + feel of the app end-to-end and should land first; the rough-edges items below build on the new shell.

1. ✅ **Full-application backdrop layer** — Persistent black backdrop wrapped at the Expo Router root in `app/_layout.tsx` via `--color-app-backdrop` token (currently `#000000`); Stack `contentStyle.backgroundColor: 'transparent'` so it shows through during route transitions. Token-wrapped so the eventual swap to gradient/lavalamp is a one-line change. Shipped 2026-05-01.

2. ⏺️ **Splash screen + app-launch animation** — App-launch sequence with the Audri wordmark animating from `Audri` → `Ai` by collapsing the middle letters (`udr` snap closed inward). Acts as the in-app splash bridge from `expo-splash-screen` (static asset) to the auth/home gate. Sequence: native splash → JS-side animation begins immediately on mount → resolves to the destination route. Requires choreographed Reanimated values; budget ~1–1.5s including settle.
   - 2b. ✅ **Iterate on app logo font** — Locked **Comfortaa** (regular, 400) as the wordmark across auth + home + animation. Replaced Sniglet. Decision context: evaluated Quicksand, Comfortaa, Mulish, Nunito, Be Vietnam Pro, Lexend on a debug route + on the live auth/home screens via a shared font-toggle. Final shortlist Comfortaa vs Nunito; Comfortaa won on the standalone `Audri` resting state. Shipped 2026-05-01.

3. ⏺️ **Auth screen redesign** — Update the sign-in screen design. Should sit on top of the new full-app backdrop and feel continuous with the launch animation's resolved logo state. Apple sign-in is still backlog; Google-only at v0.1.1.

4. ⏺️ **Onboarding screen redesign** — Replace the current verbose onboarding intro with a minimal lavalamp-animated surface. No superfluous text; the screen is just the call button. After a few seconds (timeout TBD, ~3–5s) without user interaction, a subtle `Tap to Start` hint fades in. The intent is to let the visual draw the user in — onboarding doesn't need to *explain* itself before the call begins; the agent's self-introduction handles that.

5. ⏺️ **Home screen update** — Three sub-items:
   - **Lava lamp animation** — replace the current static gradient with the same lavalamp animation used on onboarding (consistent ambient surface across the pre-call screens).
   - **Reorder plugin tiles** — current order is incidental; settle on a deliberate order based on user-facing priority (TBD at design time; likely Wiki / Todos / Research / Profile, but to be decided in the design pass).
   - **Iterate on plugin icons** — current icons are placeholder-quality; design pass on each.

6. ⏺️ **Reusable base components** — Three new primitives that consolidate the pressable-surface pattern:
   - **`GlassButton`** — pressable surface using the Liquid Glass Expo plugin where available, with a graceful fallback (likely a tinted blurred surface or solid token color) on platforms / OS versions where the plugin isn't supported. Centralizes the press feedback, haptics, and accessibility props.
   - **`CallButton`** — extends `GlassButton`. Used on onboarding, home (phone FAB), and any future contextual-call entry points. Replaces ad-hoc styling currently duplicated across screens.
   - **`TileButton`** — extends `GlassButton`. Used by the home plugin grid (and likely by future tile-style surfaces). Replaces the ad-hoc tile rendering in `PluginTile.tsx`.

7. ⏺️ **Tile-launch animation iteration** — The scale-from-tile transition into a plugin overlay landed in slice 7 but wants polish. Tighten the timing curves, ensure the source tile's position is captured precisely, and avoid the frame-1 white flash (the backdrop-layer item above is a prerequisite for fixing this cleanly).

8. ⏺️ **Plugin content padding convention** — Every plugin overlay should reserve `safeAreaInsets.bottom` worth of padding at the bottom of its scroll content. The visual treatment is: content extends edge-to-edge horizontally and visually through the bottom edge of the device, but the *scroll area* always extends past the last content row by the safe-area amount, so the home indicator never overlaps the final row of content. Codify this as a wrapper component or a hook used by every plugin overlay's root list/scroll.

9. ⏺️ **Call screen — Orb Shader animation** — Replace the current Orb component (`components/Orb.tsx`) with a shader-driven animation. Should respond to call state: idle (calm) → user speaking (subtle pulse) → agent speaking (more active) → interrupted/error (distinct visual). React-native-skia or a Reanimated-driven canvas are likely candidates; pick at design time. Pairs with the new backdrop layer — the orb floats over it.

10. ⏺️ **PluginOverlay base styling + layout iteration** — `PluginOverlay.tsx` is the shell every plugin renders inside. Pass over: header treatment (back row + title alignment), content padding (item 8 lives here), shadow / blur layer behind the overlay, dismiss-handle visual, and corner radii. End state: every plugin overlay feels like the same well-tuned shell with content swapped in.

### Plugin overlays + navigation

- ⏺️ **Gesture priority: plugin-dismiss vs system home-indicator** (P1, S) — When a plugin overlay is open (Wiki, Research, Profile, Todos), the iOS home-indicator swipe-up should **dismiss the overlay first**, not background the entire app. Only when the home screen itself is foregrounded (no overlay open, not in a call) should the system swipe-up gesture work normally. Implementation: `UIScreenEdgesDeferringSystemGestures` (iOS) — RN exposes via `useScreenEdgesDeferringSystemGestures` from `react-native-screens` or via Expo config. Enable on screens where we want swipe-up to be ours; disable on the home screen. Pairs naturally with the plugin-as-app navigation refactor — the deferring-gesture flag lives on the overlay shell. Android-side: `onBackPressed` already handles the back gesture, but pull-down-to-close is iOS-only so this is mostly an iOS concern. Source: post-slice-9 UX review.
- ⏺️ **Pending-artifact placeholders in plugin overlays** (P0, M) — All artifact UIs (Research today; Podcast / Email-draft / Brief in V1+) should show in-flight artifacts as pending entries — not invisible-until-complete. User taps the Research tile mid-generation and sees "Researching: Italian restaurants in lower Manhattan… (~2 min)" with a spinner; row hydrates to the full output when ready. Big confidence win — proves the system is working without requiring users to wait blind. Generic pattern: any plugin overlay reads BOTH the artifact collection AND a "pending tasks" view (agent_tasks where status in ('pending','running') AND kind matches), unions them with kind-specific placeholder rendering. Requires syncing agent_tasks to mobile (currently server-only — would need RLS + realtime publication migration like research_outputs got). Source: post-slice-7 UX feedback. **Note:** the agent_tasks RLS/realtime migration is the only schema change in this cycle; everything else is client-only.

### Todos surface

- ⏺️ **Todos: surface failed agent_task state on a row** (P1, S) — `useActiveAgentTasks` filters to `pending|running` only — failed research tasks become invisible. Extend the hook (or add a `useFailedAgentTasks`) and render a red error indicator + "tap to retry" affordance on the corresponding todo row. Wires into the existing `POST /calls/:id/retry-ingest` and a new `POST /tasks/:id/retry` (latter doesn't exist yet — small endpoint to add). Source: post-slice-8 punt.
- ⏺️ **Failed-ingestion retry button (UI)** (P1, S) — Server endpoint already exists (`POST /calls/:id/retry-ingest`). Needs a UI trigger. Originally scoped to land on a call-history surface in slice 8; that surface didn't make it. For v0.1.1, surface inline on the relevant row (todo / future call-history) rather than waiting on activity-stream UI. Source: slice 6.5 punt → slice 8 punt.
- ⏺️ **Todos: sub-tasks via hierarchy (rendering)** (P2, S) — Schema already supports nested todos (`parent_page_id` on a todo can point at another todo, not just a bucket). UI doesn't render hierarchy yet — todos appear flat under their bucket. Add indent-rendering when a todo's parent is another todo. Low urgency: agent-spawned todos always land directly under a bucket; this only matters for manual-create nesting. Source: post-slice-8 punt.

### Wiki section editor

- ⏺️ **WYSIWYG editor choice + replacement of plain-markdown editor** (P0, M) — **Current state (post-MVP):** `WikiSectionEditor` is a bare RN `TextInput` with monospace fallback that holds raw markdown. Functional but unfriendly — users have to know markdown syntax, and the editor doesn't render bold/italic/lists/links inline as they type. **v0.1.1 target:** WYSIWYG section editor with the basic familiar affordances (bold, italic, lists, links, inline code, headings). Doesn't need to be feature-rich — just feel native + familiar. Candidates: Lexical (Meta's, RN-friendly), TipTap (ProseMirror-based, strong web-RN parity story), or a smaller RN-native editor library (`react-native-pell-rich-editor` is the lightest option, less polish). Storage layer stays markdown — only the input affordance changes. Source: §19 + post-slice-9 review. **Note:** this is the largest item in the cycle (M effort); if it ends up dominating the patch's scope, consider promoting to its own 0.1.x slot and shipping the rest of v0.1.1 first.

## Demo target

A returning TestFlight user who used 0.1.0 should immediately notice:

1. The home grid feels "alive" — badges show new activity, the greeting reflects what's running.
2. Tapping the avatar opens settings (no more dead-end stub).
3. Plugin overlays dismiss with a swipe-up instead of accidentally backgrounding the app.
4. A research task in flight shows up in the Research overlay with a spinner instead of vanishing until done.
5. A failed research task surfaces visibly on its todo row with a retry button.
6. Editing a wiki section feels like editing rich text, not raw markdown.

If those six are true, the patch shipped.
