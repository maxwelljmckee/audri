# Build plan — v0.1.1 (UX polish + prompt engineering)

A patch-version cycle between MVP (0.1.0, shipped 2026-04-29) and the next feature wave. **Two parallel work streams:** (a) mobile-side UX polish — the rough edges that became visible once the app was usable end-to-end on TestFlight; (b) prompt-engineering tranche — addresses gaps revealed by the first real-world test calls (see "Prompt engineering tranche" section below). Both ship together as 0.1.1.

The UX cycle splits into two passes: a **visual-identity + core surfaces** pass (the priority items below — splash, auth, onboarding, home, call, plugin shell), and a **rough-edges** pass (the smaller backlog-derived items pulled forward into this patch).

Items below are copied from `backlog.md` so this doc is self-contained for the cycle. When an item ships, mark it ✅ here AND remove (or strikethrough + mark "shipped in 0.1.1") in the backlog. If we change our minds on scope mid-cycle, items can move back to `backlog.md` freely.

**Legend**

- ✅ — done
- ✴️ — partial / in progress
- ⏺️ — open / not started
- ⛔ — blocked

The top-priority block. These items reshape the look + feel of the app end-to-end and should land first; the rough-edges items below build on the new shell.

1. ✅ **Full-application backdrop layer** — Replaced the original black-token backdrop with a global `<LavaLamp>` (Animate React Native, project-owned) at the Expo Router root: 5 in-family blue blobs on a desaturated azure surface (`#0c1320`-class) under a `BlurView intensity=100, tint='dark'` frost. Custom placement math keeps blob centers ~on-screen with a 20% overshoot. Stack `contentStyle.backgroundColor: 'transparent'` everywhere so transitions don't paint over it. Pause-on-overlay wired (see tradeoffs.md). Shipped 2026-05-04.

2. ✅ **Splash screen + app-launch animation** — JS-side `SplashAnimation` overlay choreographs: 200ms pre-roll (silent) → 800ms hold with paired fade-in + scale (1 → 1.08) + lift (0 → -4px) → 150ms collapse of the `udr` clipping container → 500ms settle → 300ms overlay fade-out. Native splash uses solid azure (no logo PNG) for seamless handoff. Plays once per cold-start via a zustand singleton. Shipped 2026-05-01 (iterated through 2026-05-02).
   - 2b. ✅ **Iterate on app logo font** — Locked **Comfortaa** (regular, 400) as the wordmark across auth + home + animation. Replaced Sniglet. Decision context: evaluated Quicksand, Comfortaa, Mulish, Nunito, Be Vietnam Pro, Lexend on a debug route + on the live auth/home screens via a shared font-toggle. Final shortlist Comfortaa vs Nunito; Comfortaa won on the standalone `Audri` resting state. Shipped 2026-05-01.

3. ✅ **Auth screen redesign** — Stripped the "Sign in to continue" subtitle and the wide "Continue with Google" button. Replaced with a row of 3 square `GlassButton` provider tiles (Google active; Apple + Facebook stubbed for staged backend wire-up). Comfortaa wordmark on transparent bg over the LavaLamp. Email button removed. Shipped 2026-05-02.

4. ✅ **Onboarding screen redesign** — Minimal landing: skip-link top-right, CallButton at bottom-center (pixel-pinned to home's FAB position so the post-tap transition is spatially continuous). Three guidance lines stagger-fade in on a 3s cadence (`Turn up phone volume` / `and find a quiet place` / `to take your onboarding call`), then `Tap to Start` follows. Live phase uses `<CallButton mode="end">`. Shipped 2026-05-04.

5. ✴️ **Home screen update** — Lava lamp ambient ✅ (global, not just home). Tile reorder ⏺️ open. Plugin icon iteration ⏺️ open. The two open sub-items are pixel-light and don't depend on anything else.

6. ✅ **Reusable base components** — All three shipped, plus extra polish landed along the way:
   - **`GlassButton`** ✅ — `expo-glass-effect` GlassView on iOS 26+, `BlurView` fallback elsewhere. Defensive `try/catch` around the capability check so missing native module fails to BlurView path instead of crashing. White hairline border + 4% white surface wash on the fallback path so tiles read clearly on dark bg. Tint-aware: when a `tintColor` is provided, fallback paints a colored overlay (`${tintColor}99`) on top of the BlurView so per-button branding survives the non-glass path.
   - **`CallButton`** ✅ — 80px circle, `mode='start'` → emerald-500 / `mode='end'` → rose-500 (with rotated phone icon).
   - **`TileButton`** ✅ — square aspect-ratio glass tile (used internally; `PluginTile` shim retains the origin-rect capture for the launch animation).
   - **Cross-cutting**: `Haptics.impactAsync` (Light on tap, Medium on long-press) wired into `GlassButton` once and inherited by all consumers. Both gated on the consumer actually passing the corresponding handler so untouchable buttons don't buzz.

7. ⏺️ **Tile-launch animation iteration** — The scale-from-tile transition into a plugin overlay landed in slice 7 but wants polish. Tighten the timing curves, ensure the source tile's position is captured precisely, and avoid the frame-1 white flash (resolved as a side effect of the LavaLamp + transparent contentStyle work, but the curve tuning still wants attention).

8. ⏺️ **Plugin content padding convention** — Every plugin overlay should reserve `safeAreaInsets.bottom` worth of padding at the bottom of its scroll content. The visual treatment is: content extends edge-to-edge horizontally and visually through the bottom edge of the device, but the *scroll area* always extends past the last content row by the safe-area amount, so the home indicator never overlaps the final row of content. Codify this as a wrapper component or a hook used by every plugin overlay's root list/scroll.

9. ⏺️ **Call screen — Orb Shader animation** — Replace the current Orb component (`components/Orb.tsx`) with a shader-driven animation. Should respond to call state: idle (calm) → user speaking (subtle pulse) → agent speaking (more active) → interrupted/error (distinct visual). React-native-skia or a Reanimated-driven canvas are likely candidates; pick at design time. Pairs with the new backdrop layer — the orb floats over it.

10. ✴️ **PluginOverlay base styling + layout iteration** — Translucent surface ✅ (`rgba(12, 19, 32, 0.8)` azure/80) + intensity-40 frost BlurView landed; nested PluginStack `contentStyle: 'transparent'` patched so content area inherits the same translucency. Still ⏺️: header treatment (back row + title alignment), dismiss-handle visual, corner radii pass, and the safe-area content padding from item 8.

### Plugin overlays + navigation

- ⏺️ **Gesture priority: plugin-dismiss vs system home-indicator** (P1, S) — When a plugin overlay is open (Wiki, Research, Profile, Todos), the iOS home-indicator swipe-up should **dismiss the overlay first**, not background the entire app. Only when the home screen itself is foregrounded (no overlay open, not in a call) should the system swipe-up gesture work normally. Implementation: `UIScreenEdgesDeferringSystemGestures` (iOS) — RN exposes via `useScreenEdgesDeferringSystemGestures` from `react-native-screens` or via Expo config. Enable on screens where we want swipe-up to be ours; disable on the home screen. Pairs naturally with the plugin-as-app navigation refactor — the deferring-gesture flag lives on the overlay shell. Android-side: `onBackPressed` already handles the back gesture, but pull-down-to-close is iOS-only so this is mostly an iOS concern. Source: post-slice-9 UX review.
- ⏺️ **Pending-artifact placeholders in plugin overlays** (P0, M) — All artifact UIs (Research today; Podcast / Email-draft / Brief in V1+) should show in-flight artifacts as pending entries — not invisible-until-complete. User taps the Research tile mid-generation and sees "Researching: Italian restaurants in lower Manhattan… (~2 min)" with a spinner; row hydrates to the full output when ready. Big confidence win — proves the system is working without requiring users to wait blind. Generic pattern: any plugin overlay reads BOTH the artifact collection AND a "pending tasks" view (agent_tasks where status in ('pending','running') AND kind matches), unions them with kind-specific placeholder rendering. Requires syncing agent_tasks to mobile (currently server-only — would need RLS + realtime publication migration like research_outputs got). Source: post-slice-7 UX feedback. **Note:** the agent_tasks RLS/realtime migration is the only schema change in this cycle; everything else is client-only.

### Wiki section editor

- ⏺️ **WYSIWYG editor choice + replacement of plain-markdown editor** (P0, M) — **Current state (post-MVP):** `WikiSectionEditor` is a bare RN `TextInput` with monospace fallback that holds raw markdown. Functional but unfriendly — users have to know markdown syntax, and the editor doesn't render bold/italic/lists/links inline as they type. **v0.1.1 target:** WYSIWYG section editor with the basic familiar affordances (bold, italic, lists, links, inline code, headings). Doesn't need to be feature-rich — just feel native + familiar. Candidates: Lexical (Meta's, RN-friendly), TipTap (ProseMirror-based, strong web-RN parity story), or a smaller RN-native editor library (`react-native-pell-rich-editor` is the lightest option, less polish). Storage layer stays markdown — only the input affordance changes. Source: §19 + post-slice-9 review. **Note:** this is the largest item in the cycle (M effort); if it ends up dominating the patch's scope, consider promoting to its own 0.1.x slot and shipping the rest of v0.1.1 first.

**Architectural note (turn-level resolved 2026-05-05):** the backlog flagged "consider conditional prompt routing before scoping this tranche." The **turn-level case** is now resolved by `specs/conversational-routing.md` — static prompt at `/calls/start` with the model self-routing per turn, no dynamic prompt re-composition during the call. ### Observability — Pro fan-out claim-level audit (NEW)

- ⏺️ **Claim-level audit trail for Pro fan-out** (P0, M, Observability + Data model) — Today the only artifact of Pro's reasoning is a `wiki_log` aggregate counter (`+0 pages, ~1 pages, ...`). The actual claims extracted, where each was routed, which were skipped and *why* — none of that is persisted. This made the 2026-05-02 incident hard to diagnose. Two implementation options to weigh:
   - **(a) JSON column on `call_transcripts`** — cheapest. Add `pro_fan_out_response jsonb` (or extend the existing `tool_calls` slot conceptually). Dump Pro's structured response verbatim. Queryable via jsonb operators, but unstructured.
   - **(b) Dedicated `extracted_claims` table** — richer. `(id, transcript_id, content, status [committed/skipped/routed-no-op], reason, target_page_slug, target_section_title, decided_at)`. SQL-queryable for "show me all claims Pro skipped this week" type analysis. Migrations + RLS overhead.
   Recommend (b) for long-term value — the queryable shape pays off as we tune the prompt. Either way, redact like `wiki_log` already does. Source: 2026-05-02 incident, 2026-05-02 user request.

## Demo target

A returning TestFlight user who used 0.1.0 should immediately notice:

1. The home grid feels "alive" — badges show new activity, the greeting reflects what's running.
2. Tapping the avatar opens settings (no more dead-end stub).
3. Plugin overlays dismiss with a swipe-up instead of accidentally backgrounding the app.
4. A research task in flight shows up in the Research overlay with a spinner instead of vanishing until done.
5. A failed research task surfaces visibly on its todo row with a retry button.
6. Editing a wiki section feels like editing rich text, not raw markdown.

If those six are true, the patch shipped.
