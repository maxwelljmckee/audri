# Build plan — v0.2.0 (autonomic-loop substrate)

The phase that turns Audri from a voice-captured wiki into a system that *reflects on, maintains, and proactively grows* its own knowledge of the user. Synthesizes two parallel research passes — `notes/llm-wiki-ecosystem-survey.md` (Karpathy LLM-wiki ecosystem + Paco Cantero's Mindset) and `notes/openclaw-evaluation.md` (memory-wiki, active-memory, dreaming) — into one coherent build target.

**Source of truth:** the design rationale lives in the two notes above. This doc commits to specific shipping work and flags critical decision points for resolution before coding begins.

## Phase scope

The organizing target is the **proactive gap-filling autonomic loop** (per `project_proactive_gap_filling` memory): Audri reflects on each conversation, identifies gaps in its knowledge of the user, and surfaces questions in future conversations to fill them. The loop also carries the reframed proactive-information channel — agents introducing edifying, context-aware content unprompted (not curriculum-driven concept-mastery; that's reserved for V1+ specialist/tutor agents).

The loop has four moving parts:

1. **Structured-claims substrate** — every fact carries enough metadata for contradiction-detection, freshness, and contestability. Schema invariant the rest of the phase assumes.
2. **Agent-scope ingestion does the reflection** — the existing per-persona ingestion pass extends to also emit (a) open questions and (b) info-to-share candidates per call. No separate scheduled `gap_analysis` plugin in v0.2; the per-call cadence is sufficient and avoids batch-vs-stream complexity.
3. **A queue of agent-initiated items** — pending questions and info-shares live in one table, scoped per-persona, read by the call-side prompt composer.
4. **Call-side prompt composer reads the queue** — weaves selected items into the persona prompt as natural curiosity / sharing, bounded by saturation rules.

Plus one parallel UX surface (the Agents homescreen tile) and one supporting plugin (vault-first delta research, generalizing the existing `research` plugin). Connectors (Path 1 from the OpenClaw eval) are **deferred to v0.3** — different work, different mental model, runs as a separate track when this phase is sequenced.

The phase also includes three secondary tracks that don't sit on the autonomic-loop critical path but ship together with it: **mobile reliability** (sync verification + pull-to-refresh + Wiki pending indicator — making the existing data layer trustworthy enough to anchor everything else), **onboarding script rework** (recasts the call from life-history-first to current-life-first, and seeds new capability promises — daily briefs, connectors, dreaming — that downstream phases must honor), and a **decision point on per-turn call ingestion** (a potentially significant architectural shift that interacts with the Wiki pending indicator and with the deferred mid-call write tool).

## North-star UX (pinned)

Every item below should be checked against these. Phrasing is Paco Cantero's; the vision is wholly Audri's.

1. I speak naturally, the system handles everything else.
2. Every thought, decision, and interaction is captured without friction.
3. The system sees patterns across every area of my life, not just one app.
4. Coaching and insights arrive without me asking for them.
5. It gets smarter every single day, not just bigger.

This phase primarily targets bullets **#4** (loop closes via agent-initiated items) and **#5** (claims model + delta research = real compounding). Bullets #1 and #2 are already satisfied by the existing voice surface; bullet #3 lands as a side-effect of the per-persona reflection plus future cross-persona synthesis (deferred).

## Decision points — resolve before coding

These need a Socratic pass before any of the dependent items below get scheduled. Listed in roughly the order they need to be answered.

### ❌ DP-1. Active-memory pattern (OpenClaw lift) — DENIED 2026-05-09

**What it is.** OpenClaw's `active-memory` is a blocking pre-reply sub-agent: before *every* main agent response, a retrieval pass runs against memory with bounded `timeoutMs`, configurable `queryMode` (`message` / `recent` / `full`) and `promptStyle` (`balanced` / `strict` / `recall-heavy` / `preference-only`), and injects the result as hidden system context. The main agent doesn't reply until active-memory returns or times out.

**Resolution (Max, 2026-05-09):** **denied outright** — not deferred, not partially adopted. The always-on blocking firing pattern is architecturally incompatible with voice-first realtime. Gemini Live's natural-conversation latency budget is ~200–500ms total; an always-on per-turn blocking retrieval pass would directly fight that budget every turn even with aggressive timeouts. Audri's existing model — preload at session start (one-time cost baked into the call before audio starts) + on-demand `search_wiki` tool calls (only when the model decides it needs to look something up) — is the right shape for voice. We rely on the live agent's tool-call judgment to decide when retrieval is needed, rather than imposing it on every turn.

**Don't revisit unless:** the live agent's tool-call decisions are observably wrong (skipping lookups it should make, or confabulating from training rather than checking the wiki) AND we can't fix it with prompt-level guidance. Even then, a per-turn blocking pass is the wrong response — we'd want better tool-call prompting or a cheaper hint mechanism, not active-memory.

**Closes the dependency on DP-1 in items #5 (call-side prompt composer) and #11 (Wiki pending indicator).** The queue-driven prompt composer is a narrow at-call-start injection mechanism, not a generalized active-memory pass.

### ✅ DP-2. Dreaming (weekly synthesis pass) — RESOLVED 2026-05-09: adopt as designed, deferred to v0.3

**Naming clarification.** OpenClaw's `dreaming` and the survey's `weekly_synthesis` (insight #1's third hygiene plugin) are the same thing. "Dreaming" is the user-facing brand; the underlying mechanism is weekly synthesis with a weighted scoring rubric and a user-readable diary.

**Resolution (Max, 2026-05-09):** adopt the OpenClaw design wholesale — configurable cadence (default weekly), Light / REM / Deep phase split, six-weight scoring rubric (frequency / relevance / query-diversity / recency / consolidation / conceptual-richness), "Dreaming" branding. Bundled as the first **installable plugin** (not always-on). On install, a top-level user-scoped wiki page `Dreams` is seeded. Each dream/insight produced by the pass is its own `wiki_pages` row of `kind = 'dream'`, parented under `Dreams`. Users can review each dream and either promote (move parent_page_id to a target main-wiki page — no section rewrite needed) or archive (uses the new Archive backlog feature). Default unreviewed-dream auto-archive after 2 weeks (configurable).

**Phase placement.** Deferred to **v0.3**, paired with the Connectors track. v0.3 becomes "installable plugins" as a coherent theme: connectors + dreaming + the install-flow infrastructure they share + the Archive feature dreams need. See `build-phases/v0.3.0.md`.

**Implication for v0.2's onboarding script.** The dreaming line is **cut** from the new self-introduction script (#12b) for v0.2's ship. Restored in v0.3 when dreaming actually lands. The script in v0.2 promises four capabilities (capture / research / briefs / connectors); the dreaming line returns in v0.3's onboarding-script revision.

**Open design points (resolved as MVP defaults, revisit at v0.3 spec time):**
- `kind = 'dream'` confirmed (vs. `kind = 'note'` with parent = dreams) — gives clean Dreams-plugin UI filter and makes promotion a legible state transition (`dream` → `concept`/`note`).
- Promotion target picker: Option A for MVP (agent suggests parent based on dream content + existing wiki, user confirms or overrides), Option C for V2+ (full agent-suggest + user-browse + override).
- Unreviewed dream auto-archive: configurable, default 2 weeks.
- Cadence config: configurable per-user, default weekly.

### ✅ DP-4. Agents tile UX — RESOLVED 2026-05-09

**Resolution.** Agents plugin tile lands on the home grid. Inside the plugin:

- **Top level:** list of agent cards. Each card shows avatar, name, persona label (e.g. "Assistant"). N=1 for v0.2 (research-persona Audri); designed to scale to multiple personas in V1+.
- **Tap an agent card → agent details view.** Two sections:
  - *Configurations* (stubbed placeholder for V1+ — persona settings, model preferences, etc.)
  - *Open questions* — per-persona queue of `agent_open_items`. Read-only display with snooze/dismiss affordances per item.
- **Interaction model:** passive transparency. User sees what the persona is curious about; can snooze or dismiss. **No manual question seeding** in v0.2 — that's V1+ if observed need.

Spec to draft at coding time: `specs/agents-plugin.md`.

### ✅ DP-5. `agent_open_items` schema — RESOLVED 2026-05-09: single table

**Resolution.** Single table with `kind` enum (`question | info_share`) and a unified `status` enum covering both lifecycles. Lifecycle differences are small (questions terminate on `answered`; info-shares on `surfaced`/`engaged`); same delivery mechanism, same per-persona scoping, same saturation rules. Split later if lifecycles diverge in V1+.

**Important scope clarification (added at DP-5 close):** `info_share` covers the *between-call* enrichment channel — agent-scope ingestion produces info-share candidates after a call, the queue holds them, the call-side composer delivers them at the start of the *next* call. This is **distinct from** the live agent's *in-call* proactive info-sharing behavior, which is a separate prompt-engineering directive (see item #12c below). The two coexist: the queue is for things prepared between calls; the live agent's tool-call-driven sharing is for things noticed during the call. They don't duplicate — they cover complementary moments.

### ✅ DP-6. Claim-model field placement — RESOLVED 2026-05-09: claims-first, sections inherit

**Resolution.** All five fields (`status`, `confidence`, `evidence[]`, `last_refreshed_at`, `valid_at`) authored on `extracted_claims` — the audit-trail row. `wiki_sections` carries derived `confidence` (max or mean of supporting claims) and `last_refreshed_at` (max claim refresh) as computed columns for cheap query, not separately authored. `valid_at` lives on claims only; sections aggregate without their own temporal validity. One migration adds the columns big-bang per user's earlier confirmation.

### ✅ DP-7. Prompt-engineering risk — RESOLVED 2026-05-09: stage in four steps, isolate one variable per stage

**Resolution.** Don't ship the loop end-to-end on first cut. Stage the work so each step holds everything else constant — when something feels wrong on a real call, only one prompt is in scope to blame. Order:

1. **Substrate** — claim-model migration, `agent_open_items` table, Agents tile renders an empty queue. Live agent prompt unchanged. Verify nothing broke.
2. **Manual queue + delivery** — hand-write a few queue items in the DB. Add the call-side composer that reads them and weaves into the persona prompt. Iterate the *delivery* prompt against real calls.
3. **Generator** — wire agent-scope ingestion to auto-populate. Iterate the *generator* prompt; delivery is already locked.
4. **Live-agent "teach me" directive (#12c)** — add live web search tool + author persona directive for opportunistic in-call info-sharing using both wiki and web. Iterate this last because it's the hardest to evaluate (no queue to inspect; pure in-the-moment judgment).

**Fallback.** If Stage 3 or 4 takes longer than expected, ship v0.2.0 with Stages 1–2 only and slip auto-population + live-agent autonomy to v0.2.1.

**Note.** Max deferred the full call-prompt-architecture review to *post-v0.2*. DP-7 is narrower — it's the staging plan for the v0.2-internal prompt work. The Conversational Routing refactor (#12a) consumes the same prompt-engineering bandwidth and rides with Stages 2–4 as the persona prose evolves.

### ✅ DP-8. Per-turn call ingestion — RESOLVED 2026-05-09: defer to V2+ backlog

**Resolution.** Wiki pending indicator (#11) is the right immediate UX fix; works under either ingestion model. Per-turn architecture defers to V2+. Per-turn is additive when revisited (`call_turns` is a new child table; doesn't rewrite existing ingestion). Cost/coherence modeling against a real transcript should precede any future commit.

### ❓ DP-8 (original framing — kept for reference)

**What it is.** Today, ingestion is one batch pass after the call ends: full transcript → Phase 1 (Flash retrieval) → Phase 2 (Pro fan-out) → Phase 3 (transactional commit), with a per-user FIFO Graphile queue. Per-turn would split this: each turn (or each pair of user+agent turns) flows into its own ingestion pass with much narrower context. New schema would need a `call_turns` table as a child of `call_transcripts` (turn-level rows with role, text, timestamps, optional per-turn ingestion status).

**Why it might matter.**
- *Latency:* the Wiki pending state shrinks from "post-call ingestion duration" (currently tens of seconds to minutes) to "single-turn ingestion duration." Pairs naturally with the pending indicator (item #11 below) — could obviate the need for it for steady-state turns.
- *User-felt freshness:* mid-call references in the wiki app become live. Aligns with north-star bullet #2 ("captured without friction").
- *Eventually enables* the deferred mid-call write tool — turns are already individually committed, so the question becomes "can the call-side reasoning agent see them?" rather than "should we even commit mid-call?"
- *Cheaper Pro context per pass* — narrower input, smaller token spend per pass even if more passes run total. Net cost direction not obvious without modeling.

**Why it might not.**
- *Loss of batch coherence.* Pro currently sees the *whole* call when fan-out runs — it can identify cross-turn synthesis, resolve contradictions stated five turns apart, and pick the right hierarchy when a topic emerges over multiple turns. Per-turn loses that, or has to reconstruct it via re-reading prior commits, which is its own cost.
- *Write/retract risk.* User says "I work at Anthropic" turn 1, "actually scratch that, I work at Google" turn 5. Batch sees both and writes Google. Per-turn writes Anthropic, then has to know to retract. The bi-temporal claim model (DP-6) helps but doesn't solve.
- *Cost direction is genuinely unclear* — more passes means more LLM calls, even if each is cheaper. Could be net cheaper, net more expensive, or wash. Needs modeling against a real transcript.
- *Increased schema + queue complexity.* Per-user FIFO queue assumes one job per call; per-turn means N jobs per call with intra-call ordering invariants. Graphile can do this with job-key-mode but it's a real change.
- *Hybrid options exist.* The batch-or-per-turn binary may be a false dichotomy. Hybrid options worth naming:
  - *Per-turn lightweight + post-call full pass:* per-turn does claim extraction + simple updates only; post-call full Pro fan-out still runs for synthesis, hierarchy, and tombstoning.
  - *Adaptive batch:* commit every N turns or every M seconds of conversation, not every turn.

**Resolution needed:** Socratic pass on whether this lands in v0.2 at all, and if so in what form. My read: **defer the architectural change to v0.3 or later**, ship the pending indicator (item #11) as a UX-side fix that works regardless of underlying ingestion shape, and do the cost/coherence modeling on per-turn separately. This DP closes either as "deferred" with a follow-up note, or "adopt as hybrid in v0.2" if the modeling work shows clear wins.

### Migration handling lessons
Two Postgres footguns hit during the substrate migrations, recorded here so future migrations avoid them:
- **`_deleted GENERATED ALWAYS AS (false) STORED` cannot be in a publication's column allowlist.** Postgres rejects generated columns in column lists (error 42P10). Workaround: omit `_deleted` from the allowlist; rxdb-supabase reads it from the SELECT response, not realtime.
- **`ALTER TYPE ... ADD VALUE` and `'value'::enum` casts cannot share a transaction.** Drizzle's postgres-js migrator runs all pending migrations in one shared transaction, so even split migration files trip this. Workaround: run `ALTER TYPE ... ADD VALUE 'x'` manually via Supabase SQL editor BEFORE the next `db:migrate`, with `IF NOT EXISTS` on the migration's enum-add so retries are no-ops.
