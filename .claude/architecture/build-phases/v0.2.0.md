# Build plan — v0.2.0 (autonomic-loop substrate)

The phase that turns Audri from a voice-captured wiki into a system that *reflects on, maintains, and proactively grows* its own knowledge of the user. Synthesizes two parallel research passes — `notes/llm-wiki-ecosystem-survey.md` (Karpathy LLM-wiki ecosystem + Paco Cantero's Mindset) and `notes/openclaw-evaluation.md` (memory-wiki, active-memory, dreaming) — into one coherent build target.

**Source of truth:** the design rationale lives in the two notes above. This doc commits to specific shipping work and flags critical decision points for resolution before coding begins.

## Phase scope

The organizing target is the **proactive gap-filling autonomic loop** (per `project_proactive_gap_filling` memory): Audri reflects on each conversation, identifies gaps in its knowledge of the user, and surfaces questions in future conversations to fill them. The loop also carries the reframed proactive-information channel — agents introducing edifying, context-aware content unprompted (not curriculum-driven concept-mastery; that's reserved for V1+ specialist/tutor agents).

The loop has four moving parts:

1. **Structured-claims substrate** — every fact carries enough metadata for contradiction-detection, freshness, and contestability. Schema invariant the rest of the phase assumes.
2. **Agent-scope ingestion does the reflection** — the existing per-persona ingestion pass extends to also emit (a) open questions and (b) info-to-share candidates per call. No separate scheduled `gap_analysis` plugin in v0.2; the per-call cadence is sufficient and avoids batch-vs-stream complexity.
3. **A queue of agent-initiated items** — pending questions and info-shares live in one table, scoped per-persona, read by the call-side prompt composer.
4. **Call-side prompt composer reads the queue** — weaves selected items into the persona prompt as natural curiosity / sharing, bounded by saturation rules.

Plus one parallel UX surface (the Agents homescreen tile) and one supporting plugin (vault-first delta research, generalizing the existing `research` plugin). Connectors (Path 1 from the OpenClaw eval) are **deferred to v0.3** — different work, different mental model, runs as a separate track when this phase is sequenced.

## North-star UX (pinned)

Every item below should be checked against these. Phrasing is Paco Cantero's; the vision is wholly Audri's.

1. I speak naturally, the system handles everything else.
2. Every thought, decision, and interaction is captured without friction.
3. The system sees patterns across every area of my life, not just one app.
4. Coaching and insights arrive without me asking for them.
5. It gets smarter every single day, not just bigger.

This phase primarily targets bullets **#4** (loop closes via agent-initiated items) and **#5** (claims model + delta research = real compounding). Bullets #1 and #2 are already satisfied by the existing voice surface; bullet #3 lands as a side-effect of the per-persona reflection plus future cross-persona synthesis (deferred).

## Decision points — resolve before coding

These need a Socratic pass before any of the dependent items below get scheduled. Listed in roughly the order they need to be answered.

### ❓ DP-1. Active-memory pattern (OpenClaw lift) — adopt, defer, or ignore?

**What it is.** OpenClaw's `active-memory` is a blocking pre-reply sub-agent: before the main response, a configurable retrieval pass runs against memory with bounded `timeoutMs`, configurable `queryMode` (`message` / `recent` / `full`) and `promptStyle` (`balanced` / `strict` / `recall-heavy` / `preference-only`), and injects the result as hidden system context. Audri's call-side preload + Flash retrieval is the closest analog we have today; the OpenClaw pattern *generalizes* it into a config-driven blocking pass.

**Why it might matter for v0.2.** The call-side prompt composer that reads the open-questions queue is the same plumbing that feeds retrieval into the persona prompt. If we generalize Flash into the active-memory shape *first*, gap-questions become "one more channel" through the same pass. If we don't, we end up with two parallel mechanisms.

**Why it might not.** Our Flash retrieval works. Generalizing it adds surface area before we have evidence we need the configurability. The right time may be V1+ when multiple personas demand different retrieval profiles.

**Resolution needed:** a Socratic discussion of what the existing Flash retrieval does today, where it falls short, and whether the active-memory configurability genuinely helps or is over-engineering for v0.2. Until resolved, treat the call-side prompt composer as an extension of Flash, not a replacement for it.

### ❓ DP-2. Dreaming weighted-scoring promotion (OpenClaw lift) — adopt, defer, or ignore?

**What it is.** OpenClaw's `dreaming` is opt-in background consolidation: Light / REM / Deep phases promote short-term observations into long-term memory via weighted scoring across frequency / relevance / query-diversity / recency / consolidation / conceptual-richness. They keep a `DREAMS.md` Dream Diary for human review.

**Why it might matter for v0.2.** Without explicit weighting, our future weekly-synthesis plugin will use ad-hoc heuristics. The dreaming algorithm gives us a defensible, transparent ranking model for "what gets promoted from session-level observations to wiki-level knowledge."

**Why it might not.** Weekly synthesis isn't actually in v0.2 scope; it was originally a #1 hygiene plugin in the survey but has been deferred behind the autonomic-loop work. Adopting dreaming now is solving a problem we don't have yet.

**Resolution needed:** confirm that weekly synthesis stays out of v0.2. If yes, dreaming is a v0.3+ decision and this DP closes as "deferred." If we want even a thin synthesis pass in v0.2, dreaming's algorithm is worth lifting now.

### ❓ DP-3. Dashboard shapes (OpenClaw lift) — what wiki-health surfaces do we emit?

**What it is.** OpenClaw's `memory-wiki` ships dashboards as auto-generated markdown reports: `contradictions.md`, `stale-pages.md`, `low-confidence.md`, `open-questions.md`, `person-agent-directory.md`, `relationship-graph.md`. The `wiki_health_audit` plugin (#1 from the survey) was originally going to emit dashboard-style outputs; the question is *which* dashboards, and whether they're agent-readable artifacts, user-visible UI surfaces, or both.

**Why it might matter for v0.2.** The Agents homescreen tile (committed below) needs *something* to display. Open-questions is obviously one of those things — but the others (contradictions, stale, low-confidence) require the structured-claims substrate to be in place before they can be computed, and the user-visible surface design hasn't been spec'd.

**Why it might not.** v0.2's headline is the autonomic loop, not wiki-health observability. Dashboards beyond "open questions for this persona" may belong in a later phase focused on user-facing wiki transparency.

**Resolution needed:** decide which dashboards are in scope for v0.2. My read: only **open-questions** (per persona) is load-bearing for the loop; the rest are nice-to-have observability that fits better in a later transparency-focused phase. Confirm or push back.

### ❓ DP-4. Agents homescreen tile / plugin — UX design

The pending question queue + proactive info-shares need a UX home. The user identified the Agents plugin/homescreen tile as the right surface, but the tile doesn't exist yet. Open questions: Does this tile show *all* personas, or one selected? Does the user dismiss/snooze questions here, or only see what the persona is "thinking about"? Is this a passive transparency surface (Audri shows what it's curious about) or an active edit surface (user can add/dismiss questions directly)? What's the Transparency-vs-Proactiveness balance?

**Resolution needed:** design pass + spec doc (probably `specs/agents-plugin.md`). Block on this before shipping the loop end-to-end.

### ❓ DP-5. Single `agent_open_items` table or split tables for questions vs. info-shares?

The reframed #6 (proactive info-introduction) and the gap-filling questions are both forms of agent-initiated content: same delivery mechanism, same per-persona scoping, same saturation rules. They differ in lifecycle (questions get *answered*; info-shares just get delivered or expired) and in source (questions come from gaps; info-shares come from synthesis on existing knowledge).

Two options:
- **Single table** with `kind` enum (`question | info_share`) and a `status` enum that covers both lifecycles. Cleaner, simpler, shares all infrastructure.
- **Two tables** — `agent_open_questions` and `agent_proactive_shares`. More explicit lifecycles, slightly more code but each table is simpler.

**Resolution needed:** pick one. My lean: single table with `kind` enum for v0.2 simplicity; split later if lifecycles diverge.

### ❓ DP-6. Claim-model migration scope (where does the schema live?)

User confirmed big-bang on the claim-model migration (`status`, `confidence`, `evidence[]`, `last_refreshed_at`, `valid_at`). Open: do these fields live on `extracted_claims`, `wiki_sections`, or both? Survey doc's open-question #3 floated "claims first, sections inherit when promoted." Worth confirming the exact shape before the migration is written.

**Resolution needed:** schema sketch + review. One migration, one chance to get it right; the rest of v0.2 builds on this.

### ❓ DP-7. Question-quality prompt engineering — design risk concentration

The hardest part of the loop isn't schema or scheduling; it's making questions feel like genuine curiosity, not interrogation. The agent-scope ingestion prompt that emits questions is one half (structured candidate); the call-side prompt composer that delivers them is the other (natural delivery). Both need iterative prompt engineering against real call recordings.

**Resolution needed:** budget for this. Suggest: don't ship the loop end-to-end on first cut. Ship the substrate + queue + UX surface first, populate the queue manually for a week, observe; *then* wire the agent-scope generator and the prompt composer. This separates schema risk from prompt-engineering risk.

## In scope (committed)

### Substrate

1. ⏺️ **Structured-claims schema migration (big-bang)** — depends on **DP-6**. Adopt the OpenClaw memory-wiki claim model fields (`status: 'supported' | 'contested' | 'rejected'`, `confidence`, `evidence[]`, `last_refreshed_at`) and our bi-temporal extension (`valid_at` separate from `recorded_at`). Migration generates from the schema diff; backfill defaults to `recorded_at` for existing rows. Phase 2 fan-out prompt updated to extract validity + confidence when stated. Source-of-truth field placement decided at DP-6.

2. ⏺️ **Tiered profile maturity + personCard fields on `person` pages** — derived `maturity` enum (`stub | moderate | full`) over inbound junction-row count (COG's tiering), plus structured fields from OpenClaw's `personCard` (handles, socials, timezone, ask-for, avoid-asking-for, privacy). Call-side context-loader uses tier to decide injection depth; UI surfaces tier on the person page. Generalizable to `project`, `concept`, `org` once the pattern is proven.

### The autonomic loop

3. ⏺️ **`agent_open_items` table** — depends on **DP-5**. Per-persona queue of agent-initiated content. Fields (single-table version): `id`, `user_id`, `agent_id`, `kind` (`question | info_share`), `topic`, `body_text`, `priority`, `status` (`pending | surfaced | answered | dismissed | expired`), `created_by_task_id` (the agent-scope ingestion run that produced it), `created_at`, `surfaced_at`, `resolved_at`, `cross_domain_links` (optional refs to other agent-scope items).

4. ⏺️ **Agent-scope ingestion extended to emit open items** — depends on **DP-7**. The existing per-persona ingestion pass (see `specs/agent-scope-ingestion.md`) is extended to output, alongside its current persona-private observations, two new artifact types per call: (a) candidate questions ("what does this persona want to ask next time"), (b) candidate info-shares ("what does this persona want to proactively introduce next time, given new context"). Both flow into `agent_open_items`. **No separate `gap_analysis` plugin run** — the per-call cadence drives generation; questions are tied to recent context. Cross-call/across-time synthesis comes via the persona's accumulated agent-scope wiki, not via batch reprocessing.

5. ⏺️ **Call-side prompt composer reads the queue** — depends on **DP-1**, **DP-7**. The 7-layer system prompt at `POST /calls/start` gains a new layer (or augments an existing one — exact placement TBD with DP-1) that pulls the top-K pending items for the active persona, ranked by priority + recency, and weaves them into the persona prompt with explicit instructions for natural delivery (questions as curiosity, info-shares as relevance). Saturation budget enforced in this layer: K items per call, weekly-budget-per-persona, explicit `dismissed` items excluded.

6. ⏺️ **Post-call item resolution in fan-out** — extends the existing post-call ingestion fan-out: for each pending item the persona had at call start, attempt resolution against the transcript. Questions matched by topic + answer-detection prompt → status `answered`. Info-shares matched by "did the persona deliver it" → status `surfaced` (and possibly `answered` if the user engaged). Items that linger past N calls → status `expired` (with a hygiene sweep — see #8 below). **No mid-call write tool in v0.2** (deferred trade — confirmed in the survey's headline-extension section).

### UX surface

7. ⏺️ **Agents homescreen tile / plugin** — depends on **DP-4**. New plugin module + home-screen tile per the existing `apps/mobile` plugin pattern. Surfaces the per-persona open-items queue as a Transparency expression of what each persona is "thinking about." Dismiss/snooze interactions, queue browsing, optional manual question seeding (decided in DP-4).

### Supporting plugin work

8. ⏺️ **Vault-first delta research** — survey insight #2. Refactor the existing `research` plugin handler to (a) start with a `search_wiki` pass scoped to the topic, (b) inject retrieved sections into the research prompt as `## Existing knowledge`, (c) emit gap-targeted external queries, (d) produce a delta report referencing existing `wiki_pages`/`wiki_sections` UUIDs that should be updated/contradicted, (e) feed the delta into the same fan-out → transactional commit path used by call-side ingestion. Pure prompt revision + flow change; no schema work beyond the substrate above.

9. ⏺️ **Hygiene sweep — item expiration** — minimal version of the broader `wiki_health_audit` plugin (full version deferred per DP-3). One scheduled job per user that expires `agent_open_items` past a `max_age_calls` or `max_age_days` threshold and emits a per-persona "items expired" log. Uses the existing `agent_task_dispatch` infra. The full dashboard family (contradictions, stale, low-confidence) is gated on DP-3.

### Reframe captured (no separate work item)

- **Reframed survey insight #6** — agent personas should *proactively introduce information-rich, edifying content that builds on the user's personal context and interests*, **not** linear progression through a curriculum toward concept-mastery. Curriculum framing is reserved for V1+ specialist/tutor agents. Implementation: the info-shares half of `agent_open_items` (item #3 + #4), not a separate `concept_taught` table.

### Replaced from the survey

- **Survey insight #5 (two-output rule, voiced)** — replaced. Mid-call verbal acknowledgments would be overbearing; instead, ship a single **post-call ingestion summary report** the user can review on the home screen ("From last call I added 3 sections, updated your `Anthropic` page, and answered Audri's question about your sister"). This is a UI-only surface read against `wiki_sections.created_at`/`updated_at` + `agent_open_items.resolved_at` joined to the call session — no new schema. Slot this in once the substrate is in place; doesn't block the loop.

## Out of scope / deferred

- **Concept-mastery progression** (survey #6 original framing) — V1+ specialist/tutor agents only.
- **Mid-call write tool / voiced two-output ack** — deferred trade per survey + this doc.
- **Connectors plugin family** (gogcli / notcrawl / remindctl) — Path 1 from OpenClaw eval — deferred to v0.3 or runs as a parallel track. Different mental model; doesn't compete with the loop work.
- **Active-memory generalization** of Flash retrieval — gated on DP-1.
- **Dreaming weighted-scoring promotion** — gated on DP-2.
- **Wiki-health dashboards beyond open-questions** (contradictions / stale / low-confidence / etc.) — gated on DP-3.
- **Cross-persona synthesis** (one persona reading another's agent-scope) — V1+.
- **Scheduled `gap_analysis` plugin** as a separate run — superseded by per-call agent-scope ingestion (item #4).
- **Universal `content_index` denormalized table** — explicitly rejected; typed junctions + Postgres FTS get us the same query reach (per survey doc).

## Sequencing sketch

Subject to revision after the decision points are resolved. Each step below is a runnable end-to-end demoable thing, not a sprint.

1. **Resolve decision points** — Socratic pass on DP-1 through DP-7. Update this doc with resolutions. *Block on this before any schema work.*
2. **Substrate** — claim-model migration (#1) + person-page tier/personCard (#2). One migration, prompt updates to fan-out to populate the new fields, call-side context-loader updated to read tier.
3. **Vault-first delta research** (#8) — uses the new claim model immediately; cheapest standalone validation that the substrate works.
4. **Queue + manual seeding** — `agent_open_items` table (#3), Agents homescreen tile (#7) shows the queue, no auto-population yet. Manually seed a few items per persona; observe how they look in the prompt + on the tile.
5. **Auto-population** — agent-scope ingestion extended to emit candidates (#4). Iterate on the prompt against real call recordings until candidates feel right.
6. **Loop close** — call-side prompt composer reads queue (#5) + post-call resolution (#6). Iterate prompt-engineering on natural delivery.
7. **Hygiene + summary surface** — item expiration sweep (#9) + post-call ingestion summary (#5-replacement). Polish.

If the loop end-to-end takes longer than expected, ship items 1–4 as a usable v0.2.0 (substrate + manual queue + UX surface) and let auto-population + delivery slip into v0.2.1.
