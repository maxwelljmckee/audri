# Build plan — v0.4.0 (UX/UI continuation)

A patch cycle that picks up the open UX/UI items pulled out of v0.1.1 to let that patch ship on its visual-identity + prompt-engineering core. **Renumber history**: originally drafted as v0.3.0 → renumbered to v0.2.1 (2026-05-09) when v0.3.0 was claimed by the installable-plugins major-feature theme → renumbered to v0.2.2 (2026-05-11) when the Usage-tracking cycle claimed v0.2.1 → **bumped to v0.4.0 (2026-05-12)** to make room for additional backend cycles (v0.3.0 installable plugins + possibly more) before UX continuation; product-quality first, polish later. Distinct from v0.2.0 (autonomic-loop substrate — schema + ingestion + agent-scope work); this patch is mobile-side polish + remaining surface refinement that's been backlog-creeping since slice 8.

**Source of truth carryover.** Items below were originally scoped in `build-phases/v0.1.1.md`; this is a literal lift, with status preserved. When an item ships, mark it ✅ here AND remove from `backlog.md` if it had a parallel entry. Items can move back to backlog freely if scope tightens.

**Relationship to v0.2.0.** The two phases are independent and can be sequenced or paralleled at the user's discretion. v0.2.0 is server/worker/schema-heavy; v0.4.0 is mobile-client-heavy. The only crossover is the **Agents homescreen tile** for v0.2.0's open-items queue — that work lives in v0.2.0 (per `build-phases/v0.2.0.md` item #7 + DP-4) and is *not* duplicated here.

**Connectors track no longer relevant here.** v0.2.0 originally noted that connector work could land here as a parallel track. That was reassigned 2026-05-09: connectors + dreaming + archive + installable-plugin substrate now form the v0.3.0 theme. v0.4.0 is purely UX continuation; v0.3.0 is the next major-feature phase. The two are independent — the patch can ship before, after, or alongside v0.3.0 depending on cadence.

    - **New `todos` sidecar table** (migration 0019) with `page_id` FK to wiki shell, `parent_page_id` nullable FK to ANY wiki page (project / goal / person / etc.), `status` enum, `due_date`, `completed_at`. RLS + realtime + RxDB collection wired.

### Plugin overlays + navigation

9. ⏺️ **Gesture priority: plugin-dismiss vs system home-indicator** (P1, S) — When a plugin overlay is open (Wiki, Research, Profile, Todos), the iOS home-indicator swipe-up should **dismiss the overlay first**, not background the entire app. Only when the home screen itself is foregrounded (no overlay open, not in a call) should the system swipe-up gesture work normally. Implementation: `UIScreenEdgesDeferringSystemGestures` (iOS) — RN exposes via `useScreenEdgesDeferringSystemGestures` from `react-native-screens` or via Expo config. Enable on screens where we want swipe-up to be ours; disable on the home screen. Pairs naturally with the plugin-as-app navigation refactor — the deferring-gesture flag lives on the overlay shell. Android-side: `onBackPressed` already handles the back gesture, but pull-down-to-close is iOS-only so this is mostly an iOS concern. Source: post-slice-9 UX review.

10. ⏺️ **Pending-artifact placeholders in plugin overlays** (P0, M) — All artifact UIs (Research today; Podcast / Email-draft / Brief in V1+) should show in-flight artifacts as pending entries — not invisible-until-complete. User taps the Research tile mid-generation and sees "Researching: Italian restaurants in lower Manhattan… (~2 min)" with a spinner; row hydrates to the full output when ready. Big confidence win — proves the system is working without requiring users to wait blind. Generic pattern: any plugin overlay reads BOTH the artifact collection AND a "pending tasks" view (agent_tasks where status in ('pending','running') AND kind matches), unions them with kind-specific placeholder rendering. Requires syncing agent_tasks to mobile (currently server-only — would need RLS + realtime publication migration like research_outputs got). **Schema change:** the `agent_tasks` RLS/realtime migration is the only non-client change in this cycle's UX track; mirrors the `research_outputs` work in slice 6.5. Source: post-slice-7 UX feedback.

### Todos surface

11. ⏺️ **Todos: surface failed agent_task state on a row** (P1, S) — `useActiveAgentTasks` filters to `pending|running` only — failed research tasks become invisible. Extend the hook (or add a `useFailedAgentTasks`) and render a red error indicator + "tap to retry" affordance on the corresponding todo row. Wires into the existing `POST /calls/:id/retry-ingest` and a new `POST /tasks/:id/retry` (latter doesn't exist yet — small endpoint to add). Source: post-slice-8 punt.

12. ⏺️ **Failed-ingestion retry button (UI)** (P1, S) — Server endpoint already exists (`POST /calls/:id/retry-ingest`). Needs a UI trigger. Originally scoped to land on a call-history surface in slice 8; that surface didn't make it. Surface inline on the relevant row (todo / future call-history) rather than waiting on activity-stream UI. Source: slice 6.5 punt → slice 8 punt → v0.1.1 carryover.

13. ⏺️ **Todos: sub-tasks via hierarchy (rendering)** (P2, S) — Schema already supports nested todos (`parent_page_id` on a todo can point at another todo, not just a bucket). UI doesn't render hierarchy yet — todos appear flat under their bucket. Add indent-rendering when a todo's parent is another todo. Low urgency: agent-spawned todos always land directly under a bucket; this only matters for manual-create nesting. Source: post-slice-8 punt.

### Wiki section editor

14. ⏺️ **WYSIWYG editor choice + replacement of plain-markdown editor** (P0, M) — **Current state:** `WikiSectionEditor` is a bare RN `TextInput` with monospace fallback that holds raw markdown. Functional but unfriendly — users have to know markdown syntax, and the editor doesn't render bold/italic/lists/links inline as they type. **Target:** WYSIWYG section editor with the basic familiar affordances (bold, italic, lists, links, inline code, headings). Doesn't need to be feature-rich — just feel native + familiar. Candidates: Lexical (Meta's, RN-friendly), TipTap (ProseMirror-based, strong web-RN parity story), or a smaller RN-native editor library (`react-native-pell-rich-editor` is the lightest option, less polish). Storage layer stays markdown — only the input affordance changes. Source: `specs/mobile-app.md` §19 + post-slice-9 review. **Note:** this was flagged in v0.1.1 as the cycle's largest single item; it's the primary reason this phase exists separately.

## Demo target

A returning TestFlight user from v0.1.1 should immediately notice:

1. Editing a wiki section feels like editing rich text, not raw markdown.
2. A research task in flight shows up in the Research overlay with a spinner instead of vanishing until done.
3. A failed research task surfaces visibly on its todo row with a retry button.
4. Plugin overlays dismiss with a swipe-up instead of accidentally backgrounding the app.
5. The home grid feels "alive" — badges show new activity, the greeting reflects what's running.
6. Tapping the avatar opens settings.

If those six are true, the patch shipped.

## Open questions

- **Connectors track folded in or split out?** v0.2.0 deferred OpenClaw Path 1 connector work to "v0.3 or a parallel track." If we fold it in here, this phase grows considerably (new plugin family, OAuth flows, per-connector artifact tables, mobile UI per connector). If we split, connectors land as v0.3.1 or a parallel track. Decide before sequencing.
- **Sequencing relative to v0.2.0.** Both phases are independent; either can ship first, or they can run in parallel (different work, different files, minimal merge surface). Probably worth picking one to anchor on so the dogfood loop has a clear cadence.
- **WYSIWYG editor library choice.** Lexical vs TipTap vs `react-native-pell-rich-editor` is a discrete decision that shapes the cycle's effort. Worth a focused evaluation before scheduling — wrong choice doubles the cost.
- **Versioning.** Marketing version `0.2.1` in `app.json` before first preview build. Stick on `preview` channel; promote to `production` only after dogfood.
