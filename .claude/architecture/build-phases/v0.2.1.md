# Build plan — v0.2.1 (Usage / cost-tracking dashboard)

**Status: shipped 2026-05-11.** All in-scope items closed. Spec mirror at `specs/usage.md`. Migrations 0022 (web_search enum) + 0023 (user_settings limit columns) need `pnpm db:migrate` against prod. Worker + server redeploys required to activate new writers + endpoints. Mobile rebuild required for the new Usage screen + victory-native dep.

A narrowly scoped cycle between v0.2.0's autonomic-loop ship and v0.3.0's installable-plugins phase (UX continuation since renumbered to v0.4.0). Surfaces per-user inference spend through a new Account → Usage screen — top-line monthly total, daily bar chart, category pie, plus the substrate for user-configured spending limits that downstream phases can gate against.

**Why now.** v0.2.0 added three new inference surfaces (agent-scope producer/resolver, custom live tools, googleSearch grounding) on top of the existing ones. Cost per active user is no longer obvious, and there's no surface that tells the user (or us) where their money is going. Before the next major-feature phase (v0.3.0 installable plugins, which only adds more inference surfaces), close this loop.

**Out of scope by design.** Hard server-side enforcement of spend caps (→ backlog, high-pri), notifications, per-agent breakdown UI, historical backfill, LiteLLM migration, OpenAI Realtime migration. See "Out of scope / deferred" below.

## Phase scope

The user opens Account → Usage and sees:
1. **Monthly total** at the top — clear dollar figure for the current calendar month (user's timezone).
2. **Daily bar chart** below — one bar per day of the current month, height = $ spent that day.
3. **Category pie chart** below — breakdown of the month's spend across user-facing categories.
4. **Spending limit progress bar** — if the user has set a monthly limit, shows current / limit; if no limit set, shows a "set a limit" affordance for V1+ but stays empty at v0.2.1.

User-facing categories (NOT internal `event_kind` enum values — internal kinds collapse):

| User-facing category | Underlying `event_kind`s |
|---|---|
| **Live Agent** | `call_live` + `ingestion_prefilter` + `ingestion` + `agent_scope_ingestion` + `tool_search_wiki` + `tool_fetch_page` |
| **Web Search** | `web_search` (new — googleSearch grounding) |
| **Research** | `plugin_research` |
| **Other plugins** (V1+) | new kinds per plugin as they land |

The "Live Agent silently absorbs ingestion + tool calls" framing is load-bearing: from the user's POV, the call is one experience; surfacing fan-out cost separately would confuse without informing. Ingestion runs because the user had a call; the call's cost is the conversation cost.

## North-star UX

Cost transparency without alarm. The user should be able to answer "what did I spend this month" in 3 seconds, "where is it going" in 10, and "am I close to my limit" in a glance. Not a finance dashboard — a usage thermometer.

## Decision points

### ✅ DP-1. Cost-tracking infrastructure: static pricing constants — RESOLVED 2026-05-11

LiteLLM evaluated. **Deferred** because (a) LiteLLM Proxy doesn't handle Gemini Live (WebSocket + audio modality + grounding), so a migration leaves the most expensive surface outside the abstraction → strictly worse than unified static pricing; (b) we lean on Gemini-specific features (responseSchema, googleSearch grounding, audio Live) — each is awkward through a normalizing layer; (c) the cost-tracking case alone doesn't pay for the operational complexity of a new proxy service at MVP scale.

**Decision.** Static pricing module in TypeScript, per-model constants for input/output token rates, plus a special-case `GROUNDING_COST_USD_PER_REQUEST` for googleSearch. Implementation lives in `@audri/shared` so both server and worker import the same table.

**Revisit when** (a) we actually want to swap a non-Live model (OpenAI Realtime evaluation lives below as a forward-compat consideration), (b) we need unified observability across many services, or (c) Gemini Live support lands in LiteLLM.

### ✅ DP-2. Unified ledger (no separate web-search balance) — RESOLVED 2026-05-11

googleSearch grounding becomes a `usage_events` row with `event_kind='web_search'`, `model='gemini-grounding'`, `input_tokens=0`, `output_tokens=0`, `cost_cents = credits × pricing.WEB_SEARCH_USD_PER_REQUEST`. One row per `webSearchQueries[]` entry in the grounding metadata (each query = 1 credit). No separate balance ledger.

### ✅ DP-3. Backfill — RESOLVED 2026-05-11: skip

Going forward only. v0.2.1 cutover marks "data starts here." The Usage screen shows "No data for this month yet" until inference happens against the new writers. Avoids ambiguity ("did we charge you for that old call or not?") and the partial-recovery problem.

### ✅ DP-4. Spending-limit groundwork (soft only at v0.2.1) — RESOLVED 2026-05-11

Substrate in v0.2.1: `user_settings` gains `monthly_spend_limit_cents` (nullable, default NULL = no limit) and `monthly_spend_warning_threshold` (default 0.8). Aggregation endpoint returns current spend + limit + threshold-crossed flag. Mobile renders a progress bar + soft warning banner when crossed.

**Hard enforcement promoted to backlog (high-priority, 2026-05-11)** as its own slice — would require pre-inference checks at every Live session start + every worker job, plus user-facing UX for "you've been throttled" states. Real work, real failure modes, wants field-tested usage data before tuning. See backlog entry for design points.

### ✅ DP-5. Charting library: victory-native — RESOLVED 2026-05-11

Mature, full-featured, absorbs future chart needs (Dreaming diary stats, Notes growth charts, etc.). Bundle-size tax acceptable. Locked.

### ✅ DP-6. Timezone — RESOLVED 2026-05-11: user-local default

All "month" boundaries (current month start, daily buckets) use the user's timezone from `user_settings.timezone`, falling back to UTC only if unset. Aggregation query converts `usage_events.created_at` to user-local before bucketing.

### ✅ DP-7. Gemini Live usage metadata shape — RESOLVED 2026-05-11

Investigated via `@google/genai` SDK type definitions:

1. **Where usage arrives**: `LiveServerMessage.usageMetadata` — top-level on each server message (NOT nested under `serverContent`). Same `UsageMetadata` shape used by non-Live `generateContent` responses.
2. **Audio reporting unit**: **tokens, not seconds.** `UsageMetadata.promptTokensDetails` and `responseTokensDetails` are arrays of `{ modality: AUDIO | TEXT | IMAGE | VIDEO | DOCUMENT, tokenCount }`. Live splits the total by modality so per-modality pricing applies.
3. **Tool-call I/O**: separate field `toolUsePromptTokenCount`; fold into total input at write time.
4. **Accumulation pattern**: per-message arrivals. Sum across the session into a running total; write a single `usage_events` row at session-end.

**Schema verdict: no change to `usage_events`.** The existing `input_tokens` / `output_tokens` / `cached_tokens` schema works. Audio info is encoded faithfully in `cost_cents` (modality-aware computation at write time); the integer columns carry totals for at-a-glance analytics. A `modalities jsonb` column for fine-grained reporting can land later if needed.

**Pricing module signature**:

```ts
computeCostCents(model: string, usage: UsageMetadata): string
```

Walks `promptTokensDetails` + `responseTokensDetails` for per-modality rates when present; falls back to flat input/output for non-Live models. Live model entries in the pricing table carry per-modality rates (`inputTextUsdPerMillion`, `inputAudioUsdPerMillion`, `outputTextUsdPerMillion`, `outputAudioUsdPerMillion`, plus the standard fields for non-Live).

### ❓ DP-8. Pricing-table volatility — RESOLVE AT IMPLEMENTATION

Gemini pricing changes ~every 6 months. The static pricing module needs a clear "as-of" timestamp and a process for updates. Two shapes:

- **A. Code constants.** Simplest. Updates via PR. Stale-detection is manual (developer notices Gemini blog post).
- **B. DB-backed pricing table** keyed by model + effective_from. More flexible (different pricing for different time windows = correct backfill of historical events; A overwrites and loses the historical rate).

**Lean: A for v0.2.1** (we're not backfilling, so historical rates don't matter; and most pricing changes are small). Move to B if we ever do backfill or if pricing volatility becomes a problem. Captured here so future-Max isn't surprised.

### ❓ DP-9. Per-agent breakdown UI — DECIDE AT V0.2.1 PLANNING

Schema substrate is already there (`usage_events.agent_id` FK). The spec calls for "once available, we can also show breakdown by agent" — but at MVP every user has exactly one agent (default Assistant). Per-agent UI doesn't make sense until V1+ specialist agents land. **Decision: ship the column-wiring everywhere** (writers populate `agent_id` when known), **skip the UI slice** for v0.2.1. UI lands naturally when there are multiple agents to differentiate.

### 🚩 Forward-compat: OpenAI Realtime evaluation (not blocking v0.2.1)

Max raised (2026-05-11) the possibility of migrating Live from Gemini to OpenAI Realtime-mini for stability + ~50% cost reduction. Real consideration; **not in scope for v0.2.1**.

What v0.2.1 does that's forward-compatible: pricing keyed by `model` string (provider-agnostic), `usage_events` schema is shape-neutral, category bucketing is `event_kind`-based (not vendor-based). If/when the Realtime swap happens, we add OpenAI model rows to the pricing table; the rest of the substrate works unchanged.

What v0.2.1 does NOT solve for: the Realtime swap deletes `googleSearch` grounding (OpenAI Realtime has no analog), so the "Web Search" category would need re-sourcing (Tavily / Brave / etc.) at swap time. Separate work, separate phase.

## In scope

### Substrate

1. ✅ **Static pricing module** in `@audri/shared` — `pricing.ts` exports:
   - `MODEL_PRICING: Record<string, { inputUsdPerMillion, outputUsdPerMillion, cachedInputUsdPerMillion? }>`
   - `WEB_SEARCH_USD_PER_REQUEST = 0.014` (per Gemini pricing as of 2026-05-11: $14 per 1k grounded requests)
   - `computeCostCents(model, tokens) → string` (NUMERIC-compatible decimal string)
   - `computeWebSearchCostCents(credits) → string`
   - Each pricing entry carries an `as_of` date in a comment for stale-detection
   - Initial table: every Gemini model currently in use (`gemini-2.5-flash`, `gemini-2.5-pro`, `gemini-2.5-flash-live-preview`, `gemini-3.1-pro-preview`, `gemini-grounding`)

2. ✅ **usage_events writers** at every inference path. Cost computed inline; rows inserted in the same transaction as the inference's result-write where possible. Sub-items:

   2a. ✅ **Live session (`call_live`)** — see DP-7. Capture per-session usage metadata at session-end (or accumulate per-turn). Row written by the server when the call ends, joined to `call_transcripts.id`. Resolves whether audio is reported as tokens or seconds.

   2b. ✅ **Flash candidate retrieval (`ingestion_prefilter`)** — `apps/worker/src/ingestion/flash-candidate-retrieval.ts`. Read `response.usageMetadata`; write row inside the ingestion task. `call_transcript_id` populated.

   2c. ✅ **Pro fan-out (`ingestion`)** — `apps/worker/src/ingestion/pro-fan-out.ts`. Same pattern. Largest single cost per call.

   2d. ✅ **Agent-scope Flash (`agent_scope_ingestion`)** — `apps/worker/src/ingestion/agent-scope.ts`. Same pattern.

   2e. ✅ **Research handler (`plugin_research`)** — `apps/worker/src/research/handler.ts` already writes the row but leaves `cost_cents` at 0. Add cost computation. **Bug fix as much as feature work.**

   2f. ✅ **Custom tool calls (`tool_search_wiki` / `tool_fetch_page`)** — `apps/server/src/calls/calls.controller.ts` tool endpoints. These hit Postgres only (no inference cost), so `cost_cents = 0`. Row still recorded for audit + future per-tool-frequency analytics. Optional for v0.2.1 if the writer feels noisy; can defer.

   2g. ✅ **googleSearch grounding (`web_search`)** — read `tool_calls.groundingHits` from the call_transcripts row during post-call ingestion, write one `usage_events` row per `webSearchQueries[]` entry (or one aggregated row with `cost_cents = total_credits × WEB_SEARCH_USD_PER_REQUEST`). Easier to aggregate per-call: one row per call with N credits + cost.

3. ✅ **New `web_search` enum value migration** — `ALTER TYPE usage_event_kind ADD VALUE 'web_search'`. Standalone migration; see the `ALTER TYPE ADD VALUE` + transaction-visibility footgun documented in v0.2.0.md's "Migration handling lessons" — must apply this migration *before* the next migration that uses the value as a cast.

### Spending-limit groundwork

4. ✅ **`user_settings` schema additions:**
   - `monthly_spend_limit_cents NUMERIC(12, 2) NULL` — NULL = no limit, default
   - `monthly_spend_warning_threshold REAL NOT NULL DEFAULT 0.8` — 0..1, fraction of limit at which warning fires

5. ✅ **Limit-management endpoint** — `PUT /me/spending-limit` body `{ limit_cents: number | null, threshold?: number }`. Server-side validation: threshold in (0, 1]; limit ≥ 0 or NULL.

### Aggregation

6. ✅ **Usage aggregation endpoint** — `GET /me/usage?month=YYYY-MM` (default = current month, user-timezone). Returns:
   ```ts
   {
     month: string,                          // 'YYYY-MM'
     totalCents: number,                     // sum across all event_kinds
     daily: Array<{ day: string, cents: number }>,   // 'YYYY-MM-DD' user-local
     byCategory: {
       liveAgent: number,                    // sum of 6 collapsed kinds
       webSearch: number,
       research: number,
       // future plugins fall through into a generic `other` map
       other: Record<string, number>,
     },
     limit: { cents: number | null, thresholdReached: boolean },
   }
   ```
   Implementation: one query per month, server-side GROUP BY user-local-day + event_kind, bucketed into user-facing categories in the response builder.

### Mobile UI

7. ✅ **victory-native install** — add to `apps/mobile/package.json`. Verify it works under Expo + iOS without an extra prebuild.

8. ✅ **Account → Usage screen** — new `apps/mobile/components/account/UsageScreen.tsx`. Route added to `AccountStack`. Top → bottom:
   - Month picker (default current; allow prior months once data exists; v0.2.1 ships current-month-only since we have no backfill)

   - Category pie chart (victory-native `<VictoryPie>`)
   - Bottom: small print "Live Agent includes the call itself, post-call ingestion, and search tools. Research and Web Search are billed separately." — pre-empts the question.

9. ✅ **Limit-set UI** — small modal or settings row reachable from the Usage screen (and later from a top-level Settings). Inputs: monthly limit (USD), warning threshold (slider 50–95%). Save hits endpoint #5.

## Out of scope / deferred

- **Hard server-side spend-cap enforcement** — promoted to backlog 2026-05-11, then **promoted into `build-phases/v0.3.0.md` (item #5b) on 2026-05-12** as backend work was prioritized over UX continuation. - **Per-agent breakdown UI** — schema substrate ships (`usage_events.agent_id` populated by all writers); UI surface lands when multi-agent V1+ work materializes.
- **Historical backfill** — locked: skip per DP-3.
- **LiteLLM migration** — deferred per DP-1.
- **OpenAI Realtime migration** — separate phase if pursued; v0.2.1 is forward-compatible per the 🚩 flag above.
- **Cost-per-call detail view on Chat History** — would surface the per-call breakdown if the user taps into a transcript. Natural follow-up; not in v0.2.1.

## Sequencing sketch

Step through carefully; each stage is a runnable end-to-end demoable thing.

1. **Resolve open DPs** — DP-7 (Live usage metadata shape) is the only blocking one. Quick investigation: spin up a test call, capture all `LiveServerMessage` shapes, identify where usage data lives. ~1 hour. Updates this doc with the resolution.
2. **Pricing module** (#1) — pure code, no infra. Land first so everything else can import.
3. **Worker-side writers** (#2b, 2c, 2d, 2e) — most of the inference happens here. Writers added inline in each task's existing transaction. Cost can be observed via direct SQL queries against `usage_events` before any UI exists.
4. **`web_search` enum migration** (#3) — small standalone migration.
5. **Grounding writer** (#2g) — reads tool_calls from the call_transcripts row at ingestion time, writes the row.
6. **Live session writer** (#2a) — server-side, picks up DP-7's resolved approach.
7. **Custom tool writers** (#2f) — optional; can defer past v0.2.1 if scope crunch.
8. **Spending-limit schema** (#4) — migration.
9. **Limit endpoint + aggregation endpoint** (#5 + #6) — server-side; testable via curl before mobile.
10. **victory-native install + Usage screen** (#7 + #8) — mobile UI.
11. **Limit-set UI** (#9).
12. **Spec mirror** (#10).
13. **Field-test** on real calls — verify daily-bucket math under user-timezone, category sums match transaction-level events, threshold warning fires correctly.

**Drop-dead minimum to call v0.2.1 shipped:** stages 1–10 + 13 (substrate + writers + endpoints + read-only UI). Limit-set UI (#11) can slip to a v0.2.1.x patch or roll into v0.4.0 if needed.
